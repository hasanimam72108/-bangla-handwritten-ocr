# Chapter 3: Research Methodology

## 3.1 Overview
The methodology of this research entails the end-to-end development of an OCR pipeline capable of converting offline Bengali handwritten images into digital text. The process involves meticulous dataset alignment, image preprocessing, custom tokenization logic, and the deployment of a VisionEncoderDecoder architecture within a highly optimized training loop.

## 3.2 Dataset Selection and Alignment Strategy
The primary dataset utilized is the BN-HTRd benchmark dataset, which consists of word-level and line-level image crops derived from BBC News articles. Early experiments revealed a catastrophic physical misalignment issue when sorting image files using standard directory traversal algorithms. To resolve this, a robust, programmatical alignment strategy was engineered:
1. **Ground Truth Mapping**: The dataset provides `.xlsx` files per document (e.g., `129.xlsx`) in a `ground_truths` directory. Each row contains a unique string `Id` (e.g., `129_1_19_1`, representing `doc_writer_line_word`) alongside the Bengali text.
2. **Aggregation**: A custom Python script (`prepare_data.py`) was written to parse these Excel files, group the word-level identifiers by their line prefix (`129_1_19`), concatenate the Bengali strings with spaces, and match the result directly to the target line image `129_1_19.jpg`. 
3. **Data Splitting**: The aggregated dataset yielded 13,836 valid line samples. These were randomly split into a 90% training set (12,452 samples) and a 10% validation set (1,384 samples) using a fixed random seed to ensure reproducibility.

## 3.3 Image Preprocessing
Handwritten images possess extreme aspect ratios (very wide and short) which are inherently distorted when naively resized to square dimensions expected by Vision Transformers. To combat this, a `pad_to_square` function was implemented in `preprocessing.py`. This function resizes the longest edge of the image to the target dimension (384 pixels) while maintaining the original aspect ratio, and pads the remaining space with white pixels (value 255). Further preprocessing involves converting images to RGB tensors and normalizing them using mean and standard deviation values of `[0.5, 0.5, 0.5]`.

## 3.4 Tokenization Modifications
A critical bug was identified during initial evaluations where the HuggingFace pre-trained tokenizer natively injected arbitrary white spaces between sub-word tokens during decoding (e.g., decoding `হয়েছে` as `হ য়ে ছে`). This fundamentally breaks Bengali morphology. The issue was resolved by overriding the `convert_tokens_to_string` method within the custom `BanglaGraphemeTokenizer` (and `BanglaBPETokenizer`), explicitly replacing the ` ` character generated by SentencePiece implementations to ensure contiguous Bengali output.

## 3.5 Model Architecture
The core architecture is based on HuggingFace's `VisionEncoderDecoderModel`, utilizing the `microsoft/trocr-base-handwritten` checkpoint. 
- **Encoder**: A Vision Transformer (ViT) with approximately 86 million parameters. It divides the 384x384 padded images into 16x16 patches, applying self-attention to extract visual features.
- **Decoder**: A RoBERTa-based language model with approximately 196 million parameters. It processes the encoded visual features via cross-attention mechanisms to generate text autoregressively.
- The entire model comprises ~282.9 million parameters. The encoder was frozen initially during testing but fully fine-tuned in the final training run to adapt to Bengali strokes.

## 3.6 Training Configuration and Optimization
The model was trained in a Kaggle environment utilizing a single Tesla T4 GPU (16GB VRAM). Due to strict hardware constraints, several optimizations were implemented:
- **Hyperparameters**: The model was trained for up to 150 epochs using a batch size of 8. The optimizer utilized a learning rate of `1.0e-5` (lowered from `5.0e-5` for fine-tuning), with a weight decay of 0.01 and a gradient clipping threshold of 1.0 to prevent exploding gradients.
- **Mixed Precision**: `torch.autocast(device_type="cuda", dtype=torch.float16)` was utilized extensively during both the forward pass of training and the autoregressive `model.generate` loop in validation, doubling inference speed.
- **Early Stopping and Checkpointing**: To prevent overfitting, an early stopping patience of 25 epochs was configured. Crucially, to avoid exhausting the 20GB Kaggle disk limit, the `trainer.py` script was modified to overwrite a single `last_model.pt` file instead of accumulating historical checkpoints, keeping storage footprint flat at ~9.4GB.
- **Evaluation Loop**: Validation was decoupled to run with double the training batch size to maximize VRAM usage during generation. Decoding during validation utilized beam search with `num_beams=4`.
